Gzipped Ethscription fonts in the browser

The left side points at canonical raw /data. The right side points at decoded /content, where the browser receives actual font bytes after HTTP handling.

GitHub repo

/data preserves what is on-chain. If the payload starts with gzip bytes, consumers handle it. The API can expose the intended media type in metadata, but the raw response should not claim Content-Type: font/woff2 when the bytes are actually a gzip file containing a font.

/content serves the decoded asset with web embedding headers, so @font-face can parse it. Transport-wise, Content-Encoding: gzip is still valid and can save bandwidth.
